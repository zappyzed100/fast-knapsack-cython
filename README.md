# fast-knapsack-cython
## 100秒の壁を突破する：実務制約付きナップサック問題の高速化

汎用的な数理最適化ソルバー（Cbc/CP-SAT）では100秒以上を要する、複雑な実務制約を含むナップサック問題に対して、**Cython（AOTコンパイル）による独自アルゴリズム実装**を本命として高速に実用解を探索するプロジェクトです。Numba（JITコンパイル）実装は比較・参考用として併記しています。

## 1. 解決する課題：実務制約の複雑性と規模
本プロジェクトでは、単なる「価値と重さ」の関係を超えた、複数制約を持つ大規模ナップサック問題を対象としています。

- **アイテム総数**: **2,000個**
- **カテゴリー（グループ）数**: **200種類**
- **多次元リソース制限（3種）**: 重さ・体積・電力等に見立てた3つの独立したコスト。
- **排他制約**: ベンチマークデータには高密度な排他関係を含め、探索空間を複雑化。
- **グループ制約とボーナス**: 同一グループ内の選択数に上限と加点条件を設定。

## 2. 問題の計算困難性：2^2000 の壁
本問題の難しさは、膨大な探索空間と強い制約の組み合わせにあります。
- **天文学的な組み合わせ**: 2,000個のアイテムの組み合わせは $2^{2000}$ 通りを超えます。
- **高密度な制約**: 排他関係とグループ制約が解空間を狭め、汎用ソルバーの探索を停滞させます。
- **実測データ**: 汎用ソルバー（Cbc）は100秒の制限時間内では最適性証明に至らず、本プロジェクトではタイムアウト時点の到達解と比較しています。

## 3. ベンチマーク結果 (2,000アイテム)

以下の比較では、Cbc/CP-SAT/Gecode は厳密解探索またはそれに準じる汎用ソルバー、Cython/Numba は高速な近似解探索です。また、実装ごとに制約の表現方法と探索戦略が異なるため、目的関数値と実行時間は厳密な同条件比較ではなく、実用上の到達性能比較として読む必要があります。

**本命実装（Cython）**の狙いは、Cbc のような厳密解ソルバーの最適値を短時間で上回ることではありません。単体の焼きなまし法で瞬時に初期解を生成し、その後の進化計算で数秒から10秒程度かけて解品質を段階的に改善することで、時間制約が厳しい場面でも使いやすい探索器を実現することを重視しています。目的関数値では Cbc に及ばない一方、**0.03秒で得られる初期解**から短時間で一定品質の解へ到達できる点が Cython 実装の強みです。なお、Numba 実装は参考値として表に含めています。

| :--- | ---: | ---: | :--- |

| **Cython (進化計算/AOT)** | 4,523 | 7.7738 | 近似解/満足解 |

## 4. 高速化の技術アプローチ：戦略的パラダイムシフト（Cython 実装）

### ① 「証明」を捨てて「高速探索」に特化する
汎用ソルバーの役割は「最高値であることの数学的証明」ですが、本実装ではその証明コストを切り捨て、実務的な即時性を優先しました。
- **戦略**: 厳密解法ではなく、高速な**焼きなまし法（SA）**および**ハイブリッド進化計算（GA+SA）**を採用。
- **結果**: 単体の焼きなまし法で**0.03秒級**の初期解を返し、その後の進化計算で数秒から10秒程度かけて解品質を引き上げる二段構成を実現しました。

### ② 「数式」を「ビット演算」に変換
汎用ソルバーにとって制約判定は多数の条件分岐や行列計算を伴いますが、本実装ではその一部を低レイヤーの処理に落とし込みました。
- **独自実装**: 排他関係の判定を**ビットマスク**（`unsigned long long[16]`）で高速化。
- **効率**: 違反チェックをCPUの1クロックで済む**ビット論理積（`&`）**に置換。判定コストを物理限界まで引き下げました。

### ③ 「Swap戦略」による局所解の突破
制約が厳しい問題では、1つのアイテムを入れ替えるだけでは解が改善しない「袋小路」に陥ります。
- **独自実装**: アイテムの追加時に重量制限を超えた場合、既存のアイテムを確率的に捨てて入れ替える**Swapロジック**を実装。
- **結果**: 単体 SA だけでは届きにくいスコア帯まで、短時間の追加探索で解品質を押し上げやすい構成になっています。

### ④ 並列進化計算による精度向上
- **マルチコア活用**: Cython の `prange`（OpenMP）を用いて、複数の個体を並列にSAで磨き上げるハイブリッド構成。Numba 実装も同様の並列戦略（`parallel=True`）を採用しており、参考値として掲載しています。
- **エリート保存と交叉**: 優秀な個体を残しつつ、密度の高いアイテムを優先する**貪欲交叉（Greedy Crossover）**によって解の多様性と質を維持。

## 5. 環境構築
`uv` を用いて、Python 実行環境とビルドプロセスが管理されています。

```bash
# 環境の同期
uv sync
# データの生成
uv run python scripts/generate_and_save_problem.py
# Cythonのビルド
uv run python src/solver_cython/setup.py build_ext --inplace
# Cython ソルバーの実行
uv run python scripts/solve_with_cython.py
# Numba ソルバーの実行
uv run python scripts/solve_with_numba.py
# MiniZinc ソルバーの実行
uv run python scripts/solve_with_minizinc_solvers.py
# レポートの生成（各ソルバーの実行後に実施）
uv run python scripts/generate_report.py
